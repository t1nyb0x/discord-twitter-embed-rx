# 調査結果: ダッシュボードバージョンのBotステータス表示

## 決定事項

### 1. バージョン情報の共有手段: Redis キー

- **決定**: Redis の単一キー `app:dashboard:version` にセマンティックバージョン文字列を格納する
- **根拠**: Bot とダッシュボードは既に共有 Redis インスタンスに接続済み。追加の通信経路や依存関係が不要
- **検討した代替案**:
  - HTTP API エンドポイント → ダッシュボードへの依存が増え、ダッシュボード停止時にHTTPタイムアウトが発生する
  - 環境変数 → compose.yml にハードコードが必要で、ダッシュボード更新時に手動変更が必要
  - ファイルシステム共有ボリューム → コンテナ間でボリュームを共有する必要があり、構成が複雑化する

### 2. Redis クライアントの差異

- **決定**: Bot側は `redis` パッケージ、ダッシュボード側は `ioredis` パッケージを使用している。両方が同じ Redis インスタンスに接続するため、キーの読み書きに互換性の問題はない
- **根拠**: Redis プロトコルは共通であり、クライアントライブラリの違いは API の表面のみに影響する
- **注意**: Bot側は `redis.get()` / `redis.set()` を使用。ダッシュボード側は `redis.set()` / `redis.expire()` を使用。どちらも標準的な Redis コマンド

### 3. TTL 値と更新間隔

- **決定**: TTL = 5分（300秒）、ダッシュボード側の延長間隔 = 2分（120秒）
- **根拠**:
  - Bot のステータス更新間隔が 5分（`setInterval(updateStatus, 5 * 60 * 1000)`）であるため、TTLは最低でも1回のステータス更新サイクルをカバーする必要がある
  - 延長間隔 2分は TTL の半分以下であり、正常稼働中にTTLが切れるリスクを十分に低減する
  - ダッシュボードの `startReconcileJob` が10分間隔なので、TTL延長はそれより高頻度で行う
- **検討した代替案**:
  - TTL = 10分 → ダッシュボード停止の検知が遅れる
  - TTL = 1分 → 延長処理の頻度が高くなりすぎる

### 4. 共有定数の配置

- **決定**: Redis キー名 `app:dashboard:version` を `packages/shared/src/constants.ts` に定数として追加する
- **根拠**: Bot とダッシュボードの両方が同じキー名を参照する必要があり、`@twitterrx/shared` パッケージが既にこの用途で存在する（`GuildConfig` 等）。キー名の不一致バグを防止する

### 5. ダッシュボード側の実装箇所

- **決定**: `dashboard/src/startup.ts` の `initializeApp()` 内でバージョン書き込み、`startReconcileJob` の既存 `setInterval` パターンに倣ってTTL延長ジョブを追加
- **根拠**: 既存の初期化フローに統合することで、新しいエントリーポイントを作らずに済む。`startReconcileJob` が同じパターン（起動時に開始、setInterval で定期実行）を既に採用している

### 6. Bot側の実装箇所

- **決定**: `src/index.ts` の `updateStatus()` 関数内で Redis から `app:dashboard:version` を読み取る
- **根拠**: `updateStatus` は5分間隔で呼ばれ、既に `client.guilds.cache.size` を取得している。同じタイミングで Redis から1回 GET するだけで済む
- **注意**: `updateStatus` は現在同期関数。Redis の非同期読み取りを行うため `async` に変更し、呼び出し元も対応する

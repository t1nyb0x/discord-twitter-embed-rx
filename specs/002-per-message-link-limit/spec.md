# フィーチャー仕様: 1メッセージあたりの投稿リンク処理数上限

**フィーチャーブランチ**: `002-per-message-link-limit`\
**作成日**: 2026-05-03\
**ステータス**: ドラフト\
**入力**: ユーザー説明: "1メッセージあたりの投稿リンクを処理する - 最大処理数はダッシュボードで制御"

## ユーザーシナリオとテスト _（必須）_

### ユーザーストーリー1 - 上限を超えたリンクを無視する（優先度: P1）

Discordユーザーが1つのメッセージに複数のTwitter/X URLを貼り付けた場合、
ボットは設定で指定された上限数までのみURLを処理し、残りを無視する。
上限超過時にはその旨をメッセージで通知する。

**この優先度の理由**: 無制限の処理はAPIレート制限・Discordのレート制限・
処理時間の増大を引き起こす直接的なリスクであり、安定稼働の前提条件となる。

**独立テスト**: 上限を3に設定した状態で5件のURLを含むメッセージを送信し、
3件だけEmbedが返信されること、および超過件数の通知が届くことで完全にテスト可能。

**受け入れシナリオ**:

1. **前提** 上限が3に設定されている、**操作** URLを2件含むメッセージを送信する、**期待** 2件すべてが処理されて上限超過の通知は表示されない
2. **前提** 上限が3に設定されている、**操作** URLを3件含むメッセージを送信する、**期待** 3件すべてが処理されて上限超過の通知は表示されない
3. **前提** 上限が3に設定されている、**操作** URLを5件含むメッセージを送信する、**期待** 先頭3件のみ処理され、「3件のうち2件は上限超過のため無視しました。」という通知が全URL処理完了後に1回 `reply()` で送信され、約10秒後に自動削除される

---

### ユーザーストーリー2 - ダッシュボードで上限値を変更する（優先度: P2）

Bot管理者がダッシュボードの設定画面から上限値を変更することで、
Discordサーバーごとに1メッセージあたりの最大処理URL数を調整できる。
設定の変更はBotの再起動なしにリアルタイムで反映される。

**この優先度の理由**: サーバーごとに運用ニーズが異なり、
再起動不要のGUI操作で即時反映できることが運用効率と利便性の向上に直結する。

**独立テスト**: ダッシュボードで上限値を5に変更した直後、
5件のURLを含むメッセージが全件処理されること、
6件目以降が無視されることで検証可能（Bot再起動は不要）。

**受け入れシナリオ**:

1. **前提** 上限値が設定されていない（デフォルト3が適用）、**操作** 4件のURLを送信する、**期待** 先頭3件のみ処理される
2. **前提** ダッシュボードで上限値を5に変更して保存した、**操作** 保存直後に5件のURLを送信する、**期待** Bot再起動なしで5件すべてが処理される
3. **前提** ダッシュボードで上限値を1に変更した、**操作** 3件のURLを送信する、**期待** 1件のみ処理されて2件が無視されたことが通知される

---

### エッジケース

- 上限値が0や負数に設定された場合、デフォルト値（3）にフォールバックして処理する
- 重複URLが同一メッセージに含まれる場合、重複排除後の件数に対して上限を適用する
- normalとspoilerが混在するメッセージでは、2種類の合計ではなく**それぞれの種別ごと**に上限を適用する

## 要件 _（必須）_

### 機能要件

- **FR-001**: システムは1メッセージから抽出したURLのうち、上限値を超える分を処理せずに無視しなければならない（MUST）
- **FR-002**: 上限を超えたURLが存在した場合、ボットは全URL処理完了後に1回、無視した件数を含む通知メッセージを `reply()` で送信し、約10秒後に自動削除しなければならない（MUST）
- **FR-003**: 上限値はダッシュボードのサーバー設定画面からDiscordサーバーごとに変更できなければならない（MUST）
- **FR-004**: ダッシュボードで保存した上限値は、Botの再起動なしにリアルタイムで反映されなければならない（MUST）
- **FR-005**: 上限値がダッシュボードで未設定の場合、システムはデフォルト値（3）を使用しなければならない（MUST）
- **FR-006**: 設定値が不正（0・負数・非整数）の場合、システムはデフォルト値（3）を使用しなければならない（MUST）
- **FR-007**: URL上限のカウントはnormal URLとspoiler URLをそれぞれ独立してカウントしなければならない（MUST）
- **FR-008**: システムは上限適用の結果をデバッグログに記録しなければならない（MUST）
- **FR-010**: 通知メッセージの自動削除が失敗した場合、システムはエラーをwarnレベルでログに記録し、処理を続行しなければならない。ユーザーへの追加通知は行わない（MUST）
- **FR-009**: ダッシュボードの設定画面は上限値の入力欄と入力可能な範囲（最小1・最大5）を表示しなければならない（MUST）

### キーエンティティ

- **UrlProcessingLimit**: 1メッセージあたりの最大処理URL数の設定値。デフォルト3。ダッシュボードのサーバー設定から変更可能。`GuildConfig` に `maxUrlsPerMessage` フィールドとして追加され、Redisキャッシュ（`app:guild:{id}:config`）経由でBotに配信される
- **UrlCategories**: normal（通常）とspoiler（スポイラー）の2種類のURLカテゴリ。それぞれ独立して上限が適用される
- **GuildConfig**: 既存のサーバー設定モデル（`allowAllChannels`・ホワイトリスト）に `maxUrlsPerMessage` を追加拡張する

## コンスティテューション制約チェック _（必須）_

| 原則               | このフィーチャーへの影響                                                                                                                                                                           |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| I. クリーンコード  | デフォルト値は `DEFAULT_MAX_URLS_PER_MESSAGE = 3` として定数化。マジックナンバー禁止                                                                                                               |
| II. 早期リターン   | `processUrls` で上限超過チェックを先頭のガード節として実装し、上限内の場合のみ後続処理に進む                                                                                                       |
| III. テスト容易性  | 上限値は `GuildConfig` 経由でDIし、テスト時に任意の値を注入できるよう設計する。URLスライス・件数通知は純粋関数として切り出す                                                                       |
| IV. メソッドサイズ | `processUrls` は上限チェックと通知送信を含めても30行以内に収まる見込み。オーバーした場合は通知ロジックをプライベートメソッドへ分割する                                                             |
| V. SRP             | 上限設定の読み取りは `ChannelConfigService`（core層）、上限適用ロジックは `TweetProcessor`（core層）、通知送信は `MessageHandler`（adapter層）、ダッシュボードUI と保存API は dashboard 側に閉じる |

## 成功基準 _（必須）_

### 測定可能な成果

- **SC-001**: 上限を超えるURLを含むメッセージで、上限件数を超えた分が処理されないことが100%保証される
- **SC-002**: 上限超過時の通知が、全URL処理完了後に1回 `reply()` で送信され、約10秒後に自動削除されることが毎回確実に動作する
- **SC-003**: ダッシュボードで上限値を保存後、Botの再起動なしに次のメッセージから新しい上限値が適用される
- **SC-004**: 設定値が不正・未設定な場合にもデフォルト値（3）で安定して動作し、Botがクラッシュしない
- **SC-005**: ダッシュボードの設定画面で上限値を直感的に確認・変更できる（1操作で保存完了）

## 明確化事項

### セッション 2026-05-03

- Q: 上限超過通知の内容と送信タイミングはどうするか？ → A: 全URL処理完了後に1回まとめて通知する（例: `3件のうち2件は上限超過のため無視しました。`）
- Q: ephemeralメッセージの実現方式は？（messageCreateイベントではInteraction不可） → A: `message.reply()` で通常送信し、約10秒後に自動削除する方式を採用
- Q: normal / spoiler の上限カウントは共通か独立か？ → A: 種別ごとに独立してカウント（上限3なら normal 最大3件 + spoiler 最大3件）
- Q: ダッシュボードUIで設定できる上限値の最大値は？ → A: 最大5（最小1・最大5の範囲）
- Q: 通知メッセージの自動削除が失敗した場合の挙動は？ → A: warnレベルでログに記録して無視。ユーザーへの追加通知なし
- Q: 実装の作業順序は？（dashboardはサブモジュールで別リポジトリ） → A: Dashboard側（別リポジトリ）を先にPR・マージ → TwitterRXのサブモジュール参照更新 + Bot側実装の順

## 影響範囲と作業順序

> `dashboard/` は `github.com/t1nyb0x/discord-twitter-embed-rx-dashboard` のサブモジュールであり、
> 本リポジトリとは別のPRが必要。**必ず Dashboard → Bot の順で実装・マージすること。**

### ① Dashboard リポジトリ（先行）

| 対象                               | 変更内容                                                                  |
| ---------------------------------- | ------------------------------------------------------------------------- |
| DBスキーマ（Drizzle）              | `guild_configs` テーブルに `max_urls_per_message INTEGER NULL` カラム追加 |
| マイグレーション                   | 既存レコードへの影響なし（NULL = Bot側でデフォルト3にフォールバック）     |
| `GET /api/guilds/{guildId}/config` | レスポンスに `maxUrlsPerMessage` を含める                                 |
| `PUT /api/guilds/{guildId}/config` | `maxUrlsPerMessage`（1〜5の整数 or null）を受け取り・バリデーション追加   |
| フロントエンドUI                   | 数値入力欄（範囲: 1〜5）と保存処理の追加                                  |
| `packages/shared` の `GuildConfig` | `maxUrlsPerMessage?: number` フィールド追加                               |

### ② TwitterRX リポジトリ（Dashboard マージ後）

| 対象                           | 変更内容                                                                      |
| ------------------------------ | ----------------------------------------------------------------------------- |
| サブモジュール参照             | `dashboard` のコミットハッシュを①のマージ後のHEADに更新                       |
| `MessageHandler`               | URL上限適用・通知送信・約10秒後の自動削除ロジック追加                         |
| `ChannelConfigService`         | `GuildConfig.maxUrlsPerMessage` の読み取りとデフォルトフォールバック（3）追加 |
| `RedisChannelConfigRepository` | `app:guild:{id}:config` キャッシュに `maxUrlsPerMessage` を含める             |

## 仮定

- デフォルトの上限値は3とする（一般的なTwitterスレッドの参照件数として妥当）
- normal URLとspoiler URLはそれぞれ独立してカウントする（合算しない）
- 上限超過の通知文言は日本語とする（既存の返信メッセージが日本語のため）
- 上限値の反映は既存の `GuildConfig` → Redis Pub/Sub → Bot の伝播経路を利用する（新たなリアルタイム通信基盤は不要）
- 入力可能な上限値の範囲は最小1・最大5とする（過大な処理負荷を防ぐため）
- `guild_configs` テーブルに `max_urls_per_message` カラムを追加し、既存レコードへのデフォルト値はNULL（Bot側でデフォルト3にフォールバック）とする
